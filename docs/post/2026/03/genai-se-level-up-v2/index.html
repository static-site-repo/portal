LLM 改變的不是寫程式這件事，而是重新定義「正確」的方式

William Yeh
2026-03-16
約 7067 字 - 預計閱讀 15 分鐘
tech

幾個月前，一位朋友傳給我 GitHub Copilot 的研究報告，說：「你看，完成相同任務的速度提升了 55.8%。AI 要取代工程師了。」

我當時第一個想法不是興奮，也不是恐慌，而是一個更安靜的問題：速度快了 55.8%，但那些產出，你怎麼確認它是對的？

這個問題，才是這篇文章真正想回答的。

不是「AI 會不會取代程式設計」。那是一個太粗糙、太容易讓人站隊的問題。真正值得問的是：當 LLM 進入開發流程之後，判準移到哪裡去了——什麼叫做對，由誰來決定，用什麼來驗證。

把眼前這一輪變化理解成「AI 代替人寫程式」，只抓到了表面。更準確的說法是：工程成本沒有消失，而是在流程中重新分布。 過去大量成本落在實作撰寫與低階細節處理；現在，隨著模型能高速產生候選實作，真正變貴的東西轉成規格定義、上下文管理、驗證設計、權限控制與責任治理。1 AI 沒有取消軟體工程，而是把控制面往上推了一層。

這個變化不只是抽象感受。Jain 等人的研究顯示，只靠命名清理、模組化與規劃註解，便可讓 code generation 表現最高提升 30%；甚至只用 15% 的高品質清洗資料，也能超過 100% 的原始雜亂資料。2 這說明未來更值錢的，往往不是多寫一些程式，而是把問題整理成模型能穩定理解、系統能可靠驗證的形式。

所以，本文的核心主張很簡單：軟體工程的典範確實在變，但它變的不是工程紀律的終結，而是判準的重新定位——抽象層的提升、驗證負擔的轉移，以及治理形式的重組。

一、歷史類比有啟發，但不能把 LLM 直接當成新編譯器

軟體工程史上反覆出現的一個主題，是抽象層不斷上升。

早期程式設計高度依賴機器碼與組合語言，人類必須直接處理硬體細節、暫存器配置與控制流程。高階語言與編譯器出現後，工程師的注意力才逐步從「這條指令怎麼排」轉向「這個邏輯怎麼表達」。這不只是寫法變方便，而是人機分工被重構了。3

從這個角度看，今天的 LLM 編程確實像另一輪抽象層提升。人類不再只寫語法層面的指令，而越來越多透過自然語言、規格、測試與約束來描述意圖，再由模型生成候選實作。4

但這個歷史類比只能用來說明方向，不能直接偷渡結論。

很多人看到這裡，下一步就想說：既然高階語言取代了低階語言，今天 LLM 也會像編譯器一樣，成為新的工程基礎設施。這個推論太快了。編譯器之所以能成為工程底座，不在於它很強，而在於它可預期：同樣的合法輸入，在同樣條件下，得到語意一致的輸出。

LLM 不是這種東西。它更接近一個受語境、資料分布、提示結構與推理路徑影響的機率性實作生成器。5 拿編譯器做類比，只能說「抽象層又上升了」，不能說「LLM 已經等同於編譯器」。前者幫你看清趨勢；後者則會讓你忽略最關鍵的差異：編譯器可以默認信任，LLM 不能。

這個「信任」問題，就是判準問題的起點。

二、LLM 的真正問題，不只是非決定性，而是語意層的非決定性

一個常見說法是，現代系統本來就充滿不確定性：快取未命中、垃圾回收時機、JIT 最佳化、多執行緒交錯，哪一個不是非決定性的？既然工程師早就習慣和這些東西共處，LLM 的不穩定似乎也只是另一種可管理的系統特性。

這種說法最大的問題，是沒有分清楚不確定性發生在哪一層。

傳統系統的非決定性，大多發生在執行路徑、資源調度與效能表現層面；在設計得當的前提下，語意正確性仍可透過型別系統、同步模型、測試與形式方法加以分析與約束。LLM 的不穩定則更常直接出現在語意建構本身：同樣一個提示，可能對應不同的實作策略、不同的邊界條件假設、不同的錯誤處理方式，甚至引入安全弱點或違反隱含限制。5

這不是「跑法有點抖」，而是「你得到的東西本身可能已經換了意思」。

但把話說到這裡，也不能滑向另一個極端——好像 LLM 完全不可工程化。在任務範圍受限、提示結構清楚、工具介面穩定、測試充分的條件下，模型輸出可以明顯收斂。Anthropic 對 workflow 與 agent 的區分，本身就說明了一件事：工程上真正重要的，不是抽象地問「AI 能不能用」，而是你授予了多少自主性，又用什麼外部機制把這種自主性兜住。1

所以，更合理的定位不是把 LLM 視為新一代翻譯器，而是把它視為必須被規格、測試、權限與審計包起來的候選實作產生器。它可以很強，但不能裸奔。

三、AlphaGo 與 AlphaDev 告訴我們的，不是 AI 什麼都能做，而是它在什麼條件下特別強

要理解 AI 為什麼能在某些問題上表現驚人，最好的科普入口其實不是寫程式，而是下棋。

AlphaGo 的重要性，不只是它贏了人類棋手，而是它讓很多人第一次直觀看到：只要規則固定、目標清楚、勝負可判定，而且系統可以大量自我對弈，AI 就能在封閉問題裡快速逼近甚至超過人類。6 圍棋雖然複雜，但它仍然是一個封閉世界：棋盤有限，合法動作明確，輸贏標準預先給定。這種問題的難，主要是搜尋空間巨大，不是判準模糊。

AlphaDev 其實是同一種邏輯在程式世界裡的另一種表現。它把「尋找更好的小型排序程式」表述成一個可搜尋的問題：每一步都是組合更好的指令序列，目標函數（正確性與效能）相對明確。也正因為判準清楚、評估穩定、搜尋空間可封裝，AlphaDev 才能在某些小型排序問題上找到優於既有人類基準的方案。6

AlphaGo 和 AlphaDev 的勝利，不是因為 AI 比人更懂世界，而是因為在規則封閉、回饋明確、可大量試錯的任務裡，AI 對搜尋的利用效率非常高。

這也是為什麼它們不能被直接外推到一般企業軟體工程。企業工程面對的通常不是單一且乾淨的目標函數，而是一組彼此牴觸的要求：可維護性、模組邊界、法規遵循、事故可追溯性、部署風險、團隊交接成本、長期總持有成本。這些維度很難被壓成一個簡單分數。局部效能最優，往往不等於系統治理最優。

較合理的結論不是「AI 將接管所有工程決策」，而是：不同層級的問題，適合不同程度的自動化。 問題的判準越清楚，AI 越能發揮；判準越模糊，人的判斷越不可缺。

四、代理式工程真正改變的，不是 coding 本身，而是 review 的重心

代理式工作流的真正衝擊，不在於它證明了「人可以退出」，而在於它暴露了一個更現實的問題：當模型生成候選實作的速度遠快於人工閱讀速度，傳統以逐行 diff 為核心的審查方式，開始失去可擴展性。7

過去的 code review 預設人類能逐行看、逐段比、逐點討論。這個前提在 AI 大量生成的條件下開始動搖。候選實作的產生速度，已經遠快於人工閱讀與理解速度。

這時候真正該問的，不是「還要不要 code review」，而是「未來 review 到底 review 什麼」。

較合理的答案是：review 的對象正在從程式文字，轉向規格、風險、證據與邊界。 或者換個說法：review 正在從「檢查實作是否正確」，轉向「確認判準是否被滿足、判準本身是否被正確定義」。自治權可以上升，但只能和驗證能力一起上升；否則，系統只是更快地犯錯。1

五、多輪上下文的脆弱性，正在把規格管理推成新瓶頸

如果開發越來越依賴對話、提示與逐步澄清，那麼上下文管理就不再只是使用技巧，而會成為真正的工程問題。

多輪代理式互動，不是把同一句話反覆說，而是不斷增加限制、修正假設、引用先前輸出、放棄失敗路徑。系統不只要記住內容，還要知道哪些前提已經失效，哪些推論應該撤回，哪些局部探索不應污染全局。8

現有研究已經給出相當直接的警告。當任務從單輪明確指令，變成多輪漸進探索式對話時，模型表現可能顯著下降；問題不只在上下文變長，更在於模型容易過早承諾、被先前錯誤輸出錨定，之後只能在錯的基礎上做低效修補。Context Branching 之類的方法，試圖把版本控制語意帶進 prompt 管理，透過 checkpoint、branch、switch 等方式降低探索路徑彼此污染的問題。8

LLM 的限制未必只在單次生成不夠準，也在跨輪次規格演化的不穩定。

一旦接受這個判斷，很多工程決策就會跟著變。未來關鍵能力之一，不是讓模型多生成幾輪，而是把規格、任務與對話設計成可分支、可回退、可隔離、可重組的結構。也就是說，prompt 不再只是輸入；它開始像一種新的工程工件。

六、AI 時代的測試哲學：從 Detroit 與 London 之爭，到驗證分層

如果 LLM 與代理式系統開始承擔更多實作、重構與局部迭代工作，測試策略就不能只是沿用舊習慣，而必須重新回答一個問題：測試的作用，到底只是查錯，還是同時負責定義 AI 可以怎麼改、不能怎麼改。 換句話說：測試是在表達判準。

先把兩個經典測試學派的分歧講清楚。

Detroit school（也稱 Chicago school 或 classicist TDD）關心的是：最後系統有沒有做對事。 測試應盡量從系統的外部行為與狀態變化來驗證，而不是過度關注物件之間的互動細節。9 好處是測試比較耐重構——只要外部契約沒變，內部怎麼調整，測試不必跟著大改。

London school（mockist TDD）關心的是：系統是不是用我預期的協作方式做對事。 更強調從物件協作與互動設計出發，常採取 outside-in 的開發方式，大量使用 mock 來驗證「誰應該呼叫誰、何時呼叫、帶什麼參數」。9 好處是把設計問題提早暴露，把責任分配的模糊性在測試撰寫時就逼出來。

這裡沒有誰天然正確。它們對應兩種不同的風險感知：擔心重構成本與測試脆弱性，偏向 Detroit；擔心依賴邊界不清與責任分工混亂，偏向 London。

問題在於，到了 AI 參與開發的場景，這兩派的權重會變。

在代理式重構或生成場景中，模型很可能會調整控制流程、替換內部實作、重新切分模組，甚至用一種人類原本沒想到的方式達成同樣的外部行為。如果測試過度綁定方法呼叫次序、參數傳遞路徑與物件互動細節，那麼大量本來有效、甚至更好的重構，都可能被測試判定為失敗。過度 London 化的測試，在 AI 場景裡特別容易把「設計選擇」誤寫成「功能判準」。

這正是 AI 時代需要重新審視這場舊爭論的原因。問題已經不只是「哪一派更優雅」，而是「哪一種測試更能區分：什麼是系統真正不能變的東西，什麼只是暫時的實作路徑。」

比較有建設性的結論是：兩者各自守著不同層級的邊界。

① 對 AI 需要自由探索、自由重構的區域，測試應更接近 Detroit school 的精神：盡量用公開 API、可觀察狀態、整體行為與高階不變量來定義正確性，而不要過早把內部協作方式寫死。這一層的目的，是替模型保留足夠的搜尋空間，讓它能在不破壞外部契約的前提下自主改進實作。

② 對那些真的不能亂動的區域——對外協定、錯誤處理流程、安全檢查、高成本依賴呼叫、金流與審計事件——London school 的精神仍然有價值。因為在這些地方，我們在意的本來就不只是結果對不對，也在意它是否以正確、可追溯、可受控的方式發生。測試不是多餘，而是在保護關鍵邊界。

③ AI 時代還多出一個傳統學派爭論裡較少觸及的重點：高階可驗證性。 除了狀態驗證與互動驗證之外，還要往上加入 property-based testing、differential testing 與契約導向驗證。He 等人的研究指出，以高階不變量而不是單一 I/O 範例作為驗證核心，可以明顯改善 LLM code generation 的表現。10

測試，是護城河。它的作用不只是抓錯，也是劃定邊界：AI 的搜尋空間在這裡，不能超出這裡。讓測試來說話，才是真正的判準表達。

從這個角度看，AI 時代的測試哲學最終會走向一種驗證分層：用偏 Detroit 的測試保護外部行為與高階不變量；用偏 London 的測試保護關鍵依賴、權限邊界與高成本副作用；再用 property-based 與 differential testing 補上傳統單元測試難以覆蓋的生成風險。10

七、乾淨程式碼的新解釋：從人類可維護性到模型可理解性

過去說「乾淨程式碼」，常見的理由是：降低人類工程師的理解成本，讓維護、交接與重構變得比較容易。這個說法當然沒有錯。但到了 LLM 參與開發的時代，這套論述還得再往前走一步。今天的問題已經不只是「人看不看得懂」，而是：模型能不能穩定理解你給它的程式。

這裡最容易被低估的一點是，對模型來說，程式碼不是單純的執行對象，也是上下文材料。它要根據這些材料去推斷命名意圖、模組邊界、控制流程與可重用模式。程式越雜亂，模型越難分辨什麼是核心結構，什麼只是歷史殘留；越容易把偶然寫法誤判成必要約束，把局部噪音誤當成系統規律。

有個比喻很貼切。武俠小說裡學武的人，必須先蹲馬步。看起來什麼都沒做，卻是一切功夫的前置條件。乾淨程式碼也是類似的邏輯：它不是最終目標，而是讓一切後續工作（包括模型的理解與生成）得以穩定發生的基礎條件。蹲馬步的功夫，AI 時代反而更值錢了。

Jain 等人的研究，正好把這件事做成了可以驗證的工程問題。他們關心的不是蒐集更多程式碼資料，而是反過來問：只把現有資料清理得更有結構、更易讀，模型會不會學得更好？具體做了三件事：重新命名變數、把複雜程式拆成較小的輔助函式，以及加入自然語言形式的規劃註解。 不是改演算法，而是改「程式呈現給模型的樣子」。2

結果很直接：使用清理後的資料，表現最高可提升 30%。更有意思的是，只用 15% 的高品質清洗資料，效果就能超過用 100% 的原始雜亂資料訓練的模型。對 code generation 來說，資料品質在某些情況下比資料數量更重要。2

這個發現改變了我們對工程價值的理解。過去談工程紀律，常常像是在談長期美德：今天命名清楚、模組分明，未來團隊比較好維護。這當然成立。但在 AI 時代，這些做法不只服務未來的人，也立刻服務當下的模型。你把程式整理得越清楚，模型越容易在較小的語意空間裡工作；上下文噪音越少，生成就越穩定；結構越清楚，幻覺風險和誤解成本就越低。

在 LLM 時代，乾淨程式碼正在從人類維護性的美德，變成模型可生成性的基礎設施。

下一次再談 clean code，不要只問「這樣別人看不看得懂」。還要多問一句：這樣的結構，模型能不能穩定理解、穩定延續、穩定驗證。 這不是審美問題。這是生成條件問題。

八、未來真正要競爭的，不是誰能產生程式碼，而是誰能產生可被治理的程式碼

一旦接受前面的判斷，工程問題就會改寫成另一句話：不是能不能生成 code，而是能不能生成可被安全吸收、可被驗證、可被審計的 code。

這至少包含三層要求。

第一層是執行隔離。未經驗證的 AI 輸出，不應直接跑在高權限、真實資料或可對外互動的環境中。候選實作的執行，本身就是安全決策，而不是普通的開發便利功能。11

第二層是語意審查與變更理解。當產出速度遠超過人工閱讀速度，人類不可能只靠文字 diff 理解變更影響。未來的審查機制勢必更依賴語意差異分析、結構化重構辨識、靜態檢查與契約違反提示，協助人類把注意力集中在真正改變業務邏輯、權限流向與風險邊界的地方。12

第三層是高保證驗證。對多數一般系統來說，單元測試、整合測試與模擬環境已足夠吸收大部分風險；但在安全關鍵、金流、基礎設施與高合規場景，更形式化的狀態建模、模型檢查與規格驗證方法，就可能成為必要補充。13

生成加速了，但驗證不會因此自動變簡單。很多時候，它只會變得更重要。

九、對技術樂觀主義最強版本的回答：你說得對，但你漏了更重要的一半

真正需要正面回應的樂觀派說法，不是「AI 明天就會取代工程師」這種口號，而是它的強版本：真實軟體工程 benchmark 確實在快速進步，代理式 coding 正在逼近大規模可用門檻。

這個說法不能輕率否認。SWE-bench 與其衍生評估在 2024–2025 年確實出現顯著進展，OpenAI 甚至在 2026 年指出，SWE-bench Verified 之所以不再適合作為前沿能力指標，部分正是因為分數已逼近高位，剩餘誤差開始更多反映測試設計缺陷與訓練污染，而不只是模型能力。14

很多人會把這看成一句勝利宣言。我更願意把它看成一句警報。

因為它真正說明的，不是「工程快不重要了」，而是「當模型更強時，規格、測試與 benchmark 設計反而變得更重要」。能力一旦上來，先暴露出來的不是工程不再需要，而是驗證與治理的缺口被照得更亮。能力提升，沒有讓判準退場，反而把判準的必要性放大了。

結論

記得文章開頭那位朋友的問題嗎？「速度提升了 55.8%，AI 要取代工程師了。」

我現在的回答會是：速度提升是真的。但問題本身問錯了。

真正值得問的，從來不是「AI 會不會取代程式設計」，而是：

誰在定義規格？誰在驗證結果？誰在控制權限？誰在承擔後果？

如果這四件事沒有被設計好，模型越強，風險只會越大。 如果這四件事被設計好了，模型越強，工程收益才可能真正放大。

AI 沒有取消軟體工程。它只是把軟體工程重新定位了。未來高價值的工程師，未必是最快把功能寫出來的人，而更可能是能建立一套讓模型高效探索、讓結果可驗證、讓系統可維護、讓責任可追溯的工程體系的人。

真正值得帶走的，不是「AI 很厲害」這句空話，而是一個新的工程判準：

未來軟體工程的核心，不再只是寫出程式，而是定義什麼算對、用什麼來證明、出錯時由誰負責。

這不是對工程師的威脅，而是對工程判準的重新定義。
